# Database Schema

## Overview

PostgreSQL 17 via Prisma 7 ORM using the driver-adapter pattern (`@prisma/adapter-pg` + `pg`). The schema has 11 models across 4 logical groups: **Tenant**, **Auth** (Better Auth), **Catalog**, and **Inventory**.

## Quick Reference

| Model | Purpose | Tenant-Scoped? |
|---|---|---|
| `Tenant` | Shop/tenant root | N/A |
| `User` | Auth user (Google OAuth) | Optional (`tenantId`) |
| `Session` | Better Auth session | No |
| `Account` | Better Auth OAuth account | No |
| `Verification` | Better Auth email verification | No |
| `AttributeDefinition` | Custom field definition | Yes |
| `Product` | Inventory item | Yes |
| `ProductAttributeValue` | Custom field value on a product | Via Product |
| `StockMovement` | Stock IN/OUT transaction | Yes |
| `StockTake` | Periodic stock count session | Yes |
| `StockTakeItem` | Per-product count within stock take | Via StockTake |

## Model Details

### Tenant

Root entity. Every shop creates one tenant record. All tenant-scoped models reference this via `tenantId`.

```prisma
model Tenant {
  id             String                @id @default(cuid())
  slug           String                @unique
  shopName       String
  category       String                // e.g. "liquor", "bicycle", "general"
  currency       String                @default("रू")
  createdAt      DateTime              @default(now())
  users          User[]
  products       Product[]
  attributeDefs  AttributeDefinition[]
  stockMovements StockMovement[]
  stockTakes     StockTake[]
}
```

**Fields:**
- `id` — CUID primary key
- `slug` — URL-friendly identifier (e.g., `liquor-shop`); unique, used in route `/[tenantSlug]/...`
- `shopName` — Display name in header nav
- `category` — Shop classification; displayed as a badge
- `currency` — Currency symbol; defaults to Nepali Rupee

### User

Better Auth manages this model. The only manual addition is `tenantId`.

```prisma
model User {
  id            String    @id @default(cuid())
  tenantId      String?
  tenant        Tenant?   @relation(fields: [tenantId], references: [id])
  email         String    @unique
  name          String
  emailVerified Boolean   @default(false)
  image         String?   // Google profile photo
  createdAt     DateTime  @default(now())
  updatedAt     DateTime  @updatedAt
  sessions      Session[]
  accounts      Account[]
}
```

**Notes:**
- `tenantId` is nullable — a user can exist without a tenant (before creating/joining a shop)
- When a user creates a shop via `createShop`, their `tenantId` is set on the user record
- Email is unique across all tenants (one Google account = one user)

### Session

```prisma
model Session {
  id        String   @id @default(cuid())
  userId    String
  user      User     @relation(fields: [userId], references: [id])
  token     String   @unique
  expiresAt DateTime
  ipAddress String?
  userAgent String?
  createdAt DateTime @default(now())
  updatedAt DateTime @updatedAt
}
```

**Notes:**
- Auto-generated by Better Auth
- The `token` is used by Better Auth for session cookie management
- Expired sessions are not automatically cleaned up (can be added via a cron job)

### AttributeDefinition

Each tenant defines custom product fields. These become dynamic form fields in the product create/edit forms.

```prisma
model AttributeDefinition {
  id       String                  @id @default(cuid())
  tenantId String
  tenant   Tenant                  @relation(fields: [tenantId], references: [id])
  key      String                  // programmatic key (e.g. "expiryDate")
  label    String                  // display label (e.g. "Expiry Date")
  type     String                  // "text" | "number" | "date"
  products ProductAttributeValue[]
}
```

**Constraints:**
- `@@unique([tenantId, key])` — enforced at application level (unique within a tenant)
- No cross-tenant uniqueness — two tenants can both have an `expiryDate` key

**Types:**
- `text` — free text input
- `number` — numeric input
- `date` — date picker; date values within 30 days trigger expiry alerts on the dashboard

### Product

```prisma
model Product {
  id             String                  @id @default(cuid())
  tenantId       String
  tenant         Tenant                  @relation(fields: [tenantId], references: [id])
  name           String
  sku            String
  unitPrice      Decimal
  costPrice      Decimal                 @default(0)
  quantity       Int                     @default(0)
  lowStockLimit  Int                     @default(5)
  unit           String                  @default("pcs")
  createdAt      DateTime                @default(now())
  attributes     ProductAttributeValue[]
  movements      StockMovement[]
  stockTakeItems StockTakeItem[]

  @@unique([tenantId, sku])
  @@index([tenantId])
}
```

**Constraints & Indexes:**
- `@@unique([tenantId, sku])` — SKU uniqueness within a tenant
- `@@index([tenantId])` — fast tenant-scoped queries

**Field notes:**
- `unitPrice` and `costPrice` are Prisma `Decimal` types; must be converted to `Number()` when passing to Client Components (the `SerializedProduct` type handles this)
- `quantity` is a plain integer; stock movements use `{ increment }` / `{ decrement }` for atomic updates
- `unit` — one of `pcs`, `kg`, `ltr`, `bottle`, `pack`, `box`, `bag`, `dozen`

### ProductAttributeValue

Junction table linking a product to its custom attribute values.

```prisma
model ProductAttributeValue {
  id             String              @id @default(cuid())
  productId      String
  product        Product             @relation(fields: [productId], references: [id])
  attributeDefId String
  attributeDef   AttributeDefinition @relation(fields: [attributeDefId], references: [id])
  value          String

  @@index([productId])
}
```

**Notes:**
- Cascade delete: when an `AttributeDefinition` is deleted, all associated `ProductAttributeValue` records are deleted first (in a `$transaction`)
- When a product is deleted, its attribute values are also deleted (cascaded via `$transaction`)

### StockMovement

Every stock IN or OUT is recorded here for full audit trail.

```prisma
model StockMovement {
  id        String   @id @default(cuid())
  tenantId  String
  tenant    Tenant   @relation(fields: [tenantId], references: [id])
  productId String
  product   Product  @relation(fields: [productId], references: [id])
  type      String   // "IN" | "OUT"
  quantity  Int
  note      String?
  createdAt DateTime @default(now())

  @@index([tenantId, productId])
}
```

**Notes:**
- Every movement updates the product's `quantity` in the same `$transaction`
- A movement with `type: "OUT"` checks `product.quantity >= requested quantity` before accepting
- Stock take adjustments also create stock movements with note `"Stock take adjustment (id...)"`

### StockTake

A stock take session groups product counts into a single operation.

```prisma
model StockTake {
  id          String          @id @default(cuid())
  tenantId    String
  tenant      Tenant          @relation(fields: [tenantId], references: [id])
  status      String          @default("in_progress")  // "in_progress" | "completed" | "cancelled"
  note        String?
  createdAt   DateTime        @default(now())
  completedAt DateTime?
  items       StockTakeItem[]

  @@index([tenantId])
}
```

**Status lifecycle:**
```
in_progress ──complete──► completed
in_progress ──cancel──►  cancelled
```

- Only one `in_progress` stock take can exist per tenant at a time
- `completedAt` is set to `new Date()` when status changes to `completed` or `cancelled`

### StockTakeItem

Per-product line item within a stock take.

```prisma
model StockTakeItem {
  id               String    @id @default(cuid())
  stockTakeId      String
  stockTake        StockTake @relation(fields: [stockTakeId], references: [id], onDelete: Cascade)
  productId        String
  product          Product   @relation(fields: [productId], references: [id])
  expectedQuantity Int
  countedQuantity  Int?

  @@unique([stockTakeId, productId])
  @@index([stockTakeId])
}
```

**Notes:**
- `onDelete: Cascade` — deleting a stock take deletes all its items
- `@@unique([stockTakeId, productId])` — each product appears at most once per stock take
- `countedQuantity` is nullable — null means "not yet counted"
- `expectedQuantity` is snapshot at stock take creation time (product's current quantity)
- Discrepancy = `abs(expected - counted)` — shown on the stock take detail page


## Migration Strategy

- All migrations are in `prisma/migrations/`
- Create new migration: `pnpm prisma migrate dev --name <description>`
- After editing `auth.ts`: run `pnpm dlx @better-auth/cli generate` first (syncs Better Auth schema), then `pnpm prisma migrate dev`
- No down migrations in development — use `prisma migrate reset` to start fresh
- Production: use `prisma migrate deploy` (non-interactive)

## Decimal Serialization

Prisma `Decimal` fields (`unitPrice`, `costPrice`) cannot be passed directly to Client Components. Convert using `Number()`:

```typescript
type SerializedProduct = Omit<Product, "unitPrice" | "costPrice"> & {
  unitPrice: number;
  costPrice: number;
};

const serialized: SerializedProduct = {
  ...product,
  unitPrice: Number(product.unitPrice),
  costPrice: Number(product.costPrice),
};
```
